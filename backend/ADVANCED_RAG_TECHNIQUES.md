# 高级 RAG 技术详解

## 📚 方法10: 元数据丰富化（Metadata Enrichment）

### 原理说明

在传统RAG中，我们只存储文本内容和基本信息（章节、位置等）。**元数据丰富化**是在文本分块时提取并存储更多的结构化信息，然后在检索时利用这些元数据进行**更精准的过滤和加权**。

### 工作流程

#### 1. 文本处理阶段（导入时）

```
原始文本 → 分块 → 提取元数据 → 存储到Qdrant
```

**提取的元数据示例**：

```python
chunk = {
    "content": "张三来到了客栈，遇到了李四。两人一见如故...",
    
    # 基础元数据（当前已有）
    "chapter_id": "ch001",
    "chapter_title": "第一章 初遇",
    "paragraph_index": 5,
    
    # 🔥 新增丰富元数据
    "characters": ["张三", "李四"],  # 出现的角色
    "locations": ["客栈"],  # 出现的地点
    "keywords": ["相遇", "初识", "客栈"],  # 关键词
    "scene_type": "dialogue",  # 场景类型（对话/描述/动作/心理）
    "emotional_tone": "neutral",  # 情感基调（positive/neutral/negative）
    "summary": "张三和李四在客栈初次相遇",  # chunk摘要
    "has_conflict": false,  # 是否包含冲突
    "time_reference": "傍晚",  # 时间线索
}
```

#### 2. 检索阶段（搜索时）

**传统检索**：
```
用户问题 → Embedding → 向量检索 → 返回相似chunk
```

**元数据增强检索**：
```
用户问题 → 分析意图 → 提取关键实体
           ↓
向量检索（语义相似） + 元数据过滤（精确匹配）
           ↓
结果融合（向量分数 + 元数据匹配度）
           ↓
返回更精准的结果
```

### 具体例子

**问题**: "张三和李四第一次见面是在哪里？"

**传统方法**：
- 只依赖语义向量，可能检索到任何提到"张三"和"李四"的段落
- 可能包含他们后续的互动场景（不相关）

**元数据增强方法**：
1. **意图分析**: 查找"第一次见面"的场景
2. **实体提取**: characters=["张三", "李四"]
3. **过滤条件**:
   ```python
   filter = {
       "characters": contains_all(["张三", "李四"]),  # 必须同时出现
       "scene_type": "dialogue" OR "description",  # 相关场景类型
       "paragraph_index": min(),  # 优先早期章节
   }
   ```
4. **结果**: 精确定位到初次相遇的客栈场景

### 优势

✅ **提升精准度**: 过滤掉语义相似但不相关的内容  
✅ **支持复杂查询**: "张三在客栈的所有场景"  
✅ **减少噪声**: 避免返回无关的高相似度结果  
✅ **支持分析**: 可以统计"张三出现次数"、"对话场景比例"等

### 劣势

❌ **增加计算成本**: 导入时需要额外处理  
❌ **依赖NER准确性**: 人名、地名提取可能有误  
❌ **存储增加**: 每个chunk的元数据增加约30%存储  
❌ **需要重新导入**: 已有小说必须重新处理

## 🎭 方法11: HyDE（Hypothetical Document Embeddings）

### 原理说明

HyDE的核心思想：**与其直接用问题检索，不如让LLM先生成一个"假设性答案"，然后用这个答案去检索**。

### 为什么有效？

**传统问题**：
- 问题: "张三的真实身份是什么？"
- Embedding: [0.2, 0.5, -0.3, ...]（问题的语义向量）

**传统问题的局限**：
- 问题通常很短，语义信息有限
- 问题的向量和答案的向量可能差异较大
- 例如："谁是主角？" vs "张三是一个修仙者，他是故事的主角..."

**HyDE方法**：
```
问题 → LLM生成假设答案 → Embedding假设答案 → 检索 → 用真实原文生成最终答案
```

### 工作流程

#### Step 1: 生成假设性答案

```python
prompt = """
假设你知道问题的答案，请基于常识写一个简短的答案。
不要编造具体细节，只写合理的答案形式。

问题：张三的真实身份是什么？

假设答案（50-100字）：
"""

# LLM输出
hypothetical = "张三的真实身份是一个隐藏的高手，他可能是某个门派的弟子，
               或者是某个神秘组织的成员。他的真正实力远超表面..."
```

#### Step 2: 用假设答案做检索

```python
# 不用原问题检索
embedding = embed("张三的真实身份是什么？")  ❌

# 用假设答案检索
embedding = embed(hypothetical)  ✅
```

#### Step 3: 用检索到的真实原文生成最终答案

```python
# 检索到的真实原文
real_context = [
    "张三其实是天机门的掌门...",
    "多年前，张三化名隐藏在人间...",
]

# 用真实原文生成最终答案
final_answer = generate_answer(question, real_context)
```

### 对比示例

**场景**: 小说中张三的身份在第50章才揭露

**问题**: "张三的真实身份是什么？"

**传统方法（问题向量）**：
- 可能检索到：
  - ❌ "张三是个普通书生"（第1章，语义相似但错误）
  - ❌ "张三帮助了李四"（提到张三，但不相关）
  - ✅ "张三其实是天机门掌门"（第50章，正确答案）
- **问题**: 错误答案可能排在前面

**HyDE方法（假设答案向量）**：
- 假设答案: "张三的真实身份可能是某个隐藏的高手或重要人物..."
- 这个假设答案的语义向量**更接近真实答案的表述方式**
- 更容易检索到：
  - ✅ "张三其实是天机门掌门"（语义高度匹配）
  - ✅ "多年前，张三化名隐藏"（匹配"隐藏"概念）

### 优势

✅ **提升召回率**: 假设答案的语义更接近真实答案  
✅ **减少问答语义差异**: 答案搜答案，而不是问题搜答案  
✅ **对复杂问题更有效**: 特别是需要推理的问题  
✅ **无需重新导入**: 只改变检索方式，不改变存储

### 劣势

❌ **额外LLM调用**: 每次搜索需要先生成假设答案（+500 tokens）  
❌ **可能引入偏见**: 假设答案如果方向错误会影响检索  
❌ **响应时间增加**: 多一次LLM调用（+1-2秒）  
❌ **不适合事实查询**: 如"第一章标题是什么"（直接检索更好）

## 🔄 方法12: 多路召回融合（Multi-Retrieval Fusion）

### 原理说明

**不要把鸡蛋放在一个篮子里**。同时使用多种检索策略，然后智能融合结果。

### 多路召回策略

#### 路径1: 直接语义检索
```python
query → embedding → vector_search → results_1
```

#### 路径2: 查询改写后检索
```python
query → rewrite → embedding → vector_search → results_2
```

#### 路径3: HyDE检索
```python
query → generate_hypothetical → embedding → vector_search → results_3
```

#### 路径4: 关键词检索（BM25）
```python
query → extract_keywords → keyword_search → results_4
```

#### 路径5: 元数据过滤检索
```python
query → extract_entities → vector_search + metadata_filter → results_5
```

### 融合算法：RRF（Reciprocal Rank Fusion）

传统简单合并问题：
```python
# ❌ 简单合并可能重复
all_results = results_1 + results_2 + results_3
# ❌ 简单评分加权不公平
score = 0.4 * score_1 + 0.3 * score_2 + 0.3 * score_3
```

**RRF算法**：

```python
def reciprocal_rank_fusion(results_lists, k=60):
    """
    RRF: 对于每个结果，根据它在各路径中的排名计算分数
    
    公式: RRF_score = Σ(1 / (k + rank_i))
    """
    rrf_scores = {}
    
    for results in results_lists:
        for rank, doc in enumerate(results, 1):
            doc_id = doc.id
            if doc_id not in rrf_scores:
                rrf_scores[doc_id] = 0
            # 排名越靠前，分数越高
            rrf_scores[doc_id] += 1 / (k + rank)
    
    # 按RRF分数排序
    return sorted(rrf_scores.items(), key=lambda x: x[1], reverse=True)
```

**RRF示例**：

假设一个chunk在各路径中的排名：
- 路径1(直接检索): 第3名
- 路径2(改写检索): 第1名
- 路径3(HyDE检索): 第5名
- 路径4(关键词检索): 未出现

```python
RRF_score = 1/(60+3) + 1/(60+1) + 1/(60+5) + 0
          = 0.0159 + 0.0164 + 0.0154 + 0
          = 0.0477
```

如果另一个chunk只在路径1中是第1名：
```python
RRF_score = 1/(60+1) = 0.0164
```

**第一个chunk的RRF分数更高**（多路都检索到，更可靠）

### 优势

✅ **更高召回率**: 多种策略互补，不会漏掉重要结果  
✅ **更高可靠性**: 多路都检索到的结果更可信  
✅ **鲁棒性强**: 某个策略失效不影响整体  
✅ **适应多种问题**: 不同类型问题由不同策略处理

### 劣势

❌ **计算成本高**: 多次检索，多次LLM调用  
❌ **响应时间长**: 需要等待所有路径完成  
❌ **实现复杂**: 需要协调多个检索模块  
❌ **Token消耗大**: 是单路检索的3-5倍

## 💡 我的建议

### 优先推荐：HyDE

**理由**：
1. ✅ 无需重新导入小说
2. ✅ 实现简单（1天）
3. ✅ 效果显著（特别是推理型问题）
4. ✅ Token成本可控（约+500/次）
5. ✅ 可以随时开关

**适合**：当前已有数据的项目，快速提升效果

### 中期规划：元数据丰富化

**理由**：
1. ⚠️ 需要重新导入，但效果最全面
2. ✅ 支持复杂查询（"张三在客栈的所有场景"）
3. ✅ 为后续分析功能打基础
4. ⚠️ 需要较多开发时间

**适合**：
- 数据量还不大时（容易重新导入）
- 需要支持复杂查询时
- 有充足开发时间时

### 长期规划：多路召回

**理由**：
1. ⚠️ 实现复杂，成本高
2. ✅ 效果最好，适合生产环境
3. ⚠️ 需要HyDE、元数据等基础

**适合**：
- 对准确率要求极高的生产环境
- 有充足预算和开发资源
- 作为最终优化目标
